Backup & recovery

Database backups protect against accidental data loss from bad migrations, human error, or infrastructure failures. Users, submissions, and matches are irreplaceable — treat them accordingly.

Automatic backups (Supabase Pro plan)

On the Supabase Pro plan, daily backups are taken automatically and retained for 7 days. These are full database snapshots including schema and data.

  • Location: Supabase Dashboard > Settings > Backups
  • Retention: 7 days
  • Point-in-time recovery: Available on Pro plan (to any second within the retention window)
  • Free tier: No automatic backups — you must export manually (see below)
⚠️
Check the current plan. If the production project is on the free tier, automatic backups are not available. Upgrade to Pro or implement a manual backup schedule before going live with real users.

Downloading an automatic backup

  1. Go to Supabase Dashboard > Settings > Backups
  2. Find the backup you want (listed by date)
  3. Click Restore to restore to that point, or contact Supabase support for a downloadable dump

Manual export (table-by-table)

For a quick export of critical tables before a migration:

  1. Go to Table Editor
  2. Open the table (e.g. users)
  3. Click the Export as CSV button (top right)
  4. Save the file securely — it contains personal data and must be handled according to your data retention policy

Repeat for submissions, matches, and any other critical tables.

ℹ️
CSV exports do not include schema or PostGIS geography data in a restorable format. They are useful for quick data preservation before a migration but not as a complete database backup.

Full SQL dump via pg_dump

For a proper restorable backup, use pg_dump with your Supabase database connection string:

# Get connection string from Supabase Dashboard > Settings > Database
pg_dump "postgresql://postgres:[PASSWORD]@db.tbkjealpnoriwdosvmju.supabase.co:5432/postgres" \
  --no-owner --no-acl -f backup-$(date +%Y%m%d).sql

Connection string and password: [REDACTED] — see password manager (Settings > Database > Connection string).

matchesCriticalMatch history including mutual confirmations — losing these breaks existing mutual carpool pairs

support_ticketsImportantLoss of open tickets = missed support requests

organisationsLowEasily reconstructed from known org list
blacklistLowSmall table, reconstructable from support ticket history

Recovery procedure

If you need to restore the database:

1

Assess the damage

Determine which tables were affected and what data was lost. Check the automatic backup timestamps to find a clean restore point.

2

Contact Supabase support (for automated backup restore)

Go to supabase.com/support. Request a point-in-time restore to a specific timestamp. Pro plan required. Response time varies — have your project ID ready: tbkjealpnoriwdosvmju.

3

Or restore from manual SQL dump

If you have a manual dump, restore it using psql or import the SQL through the Supabase SQL Editor.

4

Verify after restore

Run test queries to confirm row counts and data integrity. Check edge function logs to ensure everything is working correctly.

💡
Best practice: Before every production migration, download a CSV export of users, submissions, and matches. Store them in a secure location for at least 30 days. This takes 2 minutes and can save hours of recovery work.
