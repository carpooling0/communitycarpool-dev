⚡ Edge Functions & Automation

Deploying edge functions

Edge functions are deployed using the Supabase CLI — not via git push. The CLI uploads the function code directly to Supabase and makes it available at the function's HTTPS endpoint immediately after deployment.

Prerequisites

  • Node.js installed (for npm)
  • Supabase CLI installed: npm install -g supabase
  • Supabase account credentials (or a Personal Access Token)

Initial setup

# Install CLI (if not already installed)
npm install -g supabase

# Log in to Supabase
supabase login
# Opens a browser to authenticate

# Verify installation
supabase --version

Linking to a project

Before deploying, link the CLI to the target project. You must re-link when switching between dev and prod.

# Link to PRODUCTION
supabase link --project-ref tbkjealpnoriwdosvmju

# Link to DEV (replace with dev project ID)
supabase link --project-ref [DEV_PROJECT_ID]

# Confirm which project is currently linked
supabase status
⚠️
Always confirm the linked project before deploying. Running supabase functions deploy while linked to the wrong project will overwrite functions in that environment.

Deploying a single function

# Deploy one function (recommended for targeted updates)
supabase functions deploy batch-send-emails

# The function source must be in the correct directory:
# Supabse Edge Functions/batch-send-emails/index.ts
# (Note: directory has a typo — "Supabse" not "Supabase")

Deploying all functions

# Deploy every function in the functions directory
supabase functions deploy

# This deploys ALL functions — use with caution on production

Managing secrets

Secrets (environment variables accessible to edge functions via Deno.env.get()) are managed separately from deployments:

# Set a secret
supabase secrets set RESEND_API_KEY=re_xxxxxxxxxxxxxxxxxxxxxxxx

# Set multiple secrets at once
supabase secrets set KEY1=value1 KEY2=value2

# List all secrets (names only — values are never shown)
supabase secrets list

# Remove a secret
supabase secrets unset OLD_KEY_NAME
ℹ️
Secrets are not redeployed when you run supabase functions deploy. They persist independently. You only need to set a secret once per environment (dev/prod) — or when rotating it.

Recommended dev-to-prod workflow

1

Edit function locally

Make changes to Supabse Edge Functions/[function-name]/index.ts

2

Deploy to dev

supabase link --project-ref [DEV_PROJECT_ID]
supabase functions deploy [function-name]
3

Test on dev site

Use the dev site (carpooling0.github.io/communitycarpool-dev) to test end-to-end. Check Supabase logs for errors.

4

Deploy to production

supabase link --project-ref tbkjealpnoriwdosvmju
supabase functions deploy [function-name]
5

Verify on production

Check the Supabase Logs > Edge Functions view for the deployed function. Confirm no errors in the first few invocations.

Viewing edge function logs after deploy

  1. Supabase Dashboard > Logs > Edge Functions
  2. Select the function name from the dropdown
  3. Filter by time (last 30 minutes is a good default after a fresh deploy)
  4. Look for 200 responses with expected output; investigate any 4xx or 5xx entries

Cold start behaviour

Supabase edge functions run in Deno isolates. After a period of inactivity (typically a few minutes), the next request triggers a cold start — the function boots from scratch before handling the request. This can add 1–3 seconds of latency to the first request. Subsequent requests are fast.

Cold starts are normal and expected. If you see 502 errors on cold start (not just slow responses), the most common cause is module-level code that fails when secrets are missing — see the Edge functions reference for the known SES initialisation bug and fix pattern.
