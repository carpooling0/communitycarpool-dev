Dev vs Prod environments

Community Carpool runs two fully separate environments — development and production — each with its own Supabase project, GitHub repository, and configuration. Understanding which environment you are working in at any given moment is critical to avoiding data loss or unintended changes affecting real users.

⚠️
Production contains real user data. Any change to production — database, edge functions, or frontend — is live immediately. Always develop and test on dev first.

Config file (config.js)

The frontend uses config.js (in the repo root) to set environment-specific values:

// config.js — used by the live site
const SUPABASE_URL = 'https://tbkjealpnoriwdosvmju.supabase.co';
const SUPABASE_ANON_KEY = '[REDACTED]';
const GOOGLE_MAPS_API_KEY = '[REDACTED]';

The dev repo has a separate config.js pointing to the dev Supabase project. A config.dev.js file also exists locally but is not committed (it contains dev credentials). Never overwrite the prod config.js with dev values.

Testing mode in production

Production currently has testing_mode = true in the config table. This means batch-send-emails will not send emails to real users unless they have email_whitelist = true. This is a safety measure while the platform is in early launch. See Testing mode for how to disable it when ready to go live.

Promotion flow: dev to production

Follow this sequence when promoting changes from dev to production:

1

Develop & test on dev

Make changes locally, push to dev repo, test on the dev site using dev Supabase data.

2

Apply DB migrations to dev first

Run any SQL migrations on the dev Supabase project first. Verify they work correctly.

3

Deploy edge functions to dev

Link the Supabase CLI to the dev project and deploy. Test end-to-end.

supabase link --project-ref [DEV_PROJECT_ID]
supabase functions deploy function-name
4

Apply migrations to production

Once confirmed working on dev, apply the same SQL to the prod Supabase SQL Editor. Download a backup first.

5

Deploy edge functions to production

Re-link the CLI to the prod project, then deploy.

supabase link --project-ref tbkjealpnoriwdosvmju
supabase functions deploy function-name
6

Push frontend to production

Push the final code to the prod GitHub repo: git push origin main. Verify on communitycarpool.org.

Supabase CLI project linking

The Supabase CLI remembers which project it is linked to. Always confirm the link before deploying:

# Check current link
supabase status

# Re-link to dev
supabase link --project-ref [DEV_PROJECT_ID]

# Re-link to prod
supabase link --project-ref tbkjealpnoriwdosvmju
ℹ️
The CLI stores the linked project in .supabase/config.toml in your local working directory. This file should not be committed to git as it may contain project-specific references.
