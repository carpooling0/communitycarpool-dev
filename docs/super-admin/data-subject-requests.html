🚨 Compliance & Incidents

Data subject requests

Users have the right to access, erase, or rectify their personal data. Respond to these requests within 30 days. Most requests come via the support page — handle them promptly and document your actions.

Types of requests

Request typeDescriptionHow to handle
Right to erasure (most common) User wants their data deleted Preferred: user self-serves via manage-deletion flow. Admin-assisted: use admin panel Deletions, or initiate via SQL.
Right to access User wants to know what data we hold about them Look up user by email in Supabase, compile data from users + submissions + matches tables. Email a summary.
Right to rectification User wants incorrect data corrected Update via SQL Editor. Common: typo in name, wrong email, incorrect location.

What data is stored per user

  • users table: name, email, organisation code, journey token, account creation date, T&C acceptance timestamp, deletion request status
  • submissions table: from/to location text, from/to coordinates (PostGIS), work schedule, distance preference, submission date
  • matches table: match pairs (no names — only submission IDs), match strength score, status, interest timestamps
  • support_tickets table: email, message content, ticket type, dates

After deletion: the deletion_log retains the user's email and deletion timestamp for compliance purposes, but no personal content (locations, names, messages) is retained.

Handling a right to access request

-- Gather user data
SELECT u.*, s.from_location, s.to_location, s.created_at as sub_created, s.journey_status
FROM users u
LEFT JOIN submissions s ON u.user_id = s.user_id
WHERE u.email = 'requestor@example.com';

-- Gather match data
SELECT m.match_id, m.match_strength, m.status, m.created_at
FROM matches m
JOIN submissions s ON s.sub_id = m.sub_a_id OR s.sub_id = m.sub_b_id
JOIN users u ON u.user_id = s.user_id
WHERE u.email = 'requestor@example.com';

Compile the results and email them to the requesting user via the support email address. Do not share data about other users (e.g. the identity of their matches).

Handling a right to erasure request

If the user has already submitted via the self-service deletion flow, they are in the system and will be deleted after the grace period. No admin action needed — check users.deletion_requested_at.

If the user cannot use the self-service flow (e.g. no longer has access to their email), an admin can process a manual deletion:

-- Check the user exists
SELECT user_id, email, deletion_requested_at FROM users WHERE email = 'requestor@example.com';

-- Initiate deletion manually (sets deletion date to now, cron will process in next run)
UPDATE users SET deletion_requested_at = now() WHERE email = 'requestor@example.com';
UPDATE submissions SET journey_status = 'deletion_pending' WHERE user_id = [user_id];

Or process immediately (use with caution — irreversible):

-- Record in deletion_log first
INSERT INTO deletion_log (user_id, email, deleted_at, reason, submissions_deleted, matches_deleted)
SELECT u.user_id, u.email, now(), 'admin_manual_deletion',
  (SELECT COUNT(*) FROM submissions WHERE user_id = u.user_id),
  (SELECT COUNT(*) FROM matches m JOIN submissions s ON s.sub_id IN (m.sub_a_id, m.sub_b_id) WHERE s.user_id = u.user_id)
FROM users u WHERE u.email = 'requestor@example.com';

-- Delete matches, submissions, user (in order)
DELETE FROM matches WHERE sub_a_id IN (SELECT sub_id FROM submissions WHERE user_id = [user_id])
  OR sub_b_id IN (SELECT sub_id FROM submissions WHERE user_id = [user_id]);
DELETE FROM submissions WHERE user_id = [user_id];
DELETE FROM users WHERE user_id = [user_id];

Handling a right to rectification request

-- Fix a name typo
UPDATE users SET first_name = 'Correct', last_name = 'Name' WHERE email = 'user@example.com';

-- Fix a location text (coordinates cannot be manually corrected — user must resubmit)
UPDATE submissions SET from_location = 'Corrected Address, City' WHERE sub_id = [id];

Backup copies note

Supabase automatic backups retain a snapshot of the database for up to 7 days. If a user's data is deleted and they later contact you, the data may still exist in a backup snapshot within the retention window. Note this in your response if relevant — do not restore backup data without a clear legal reason to do so.

Response timeline

Best practice (and the standard under most data protection frameworks): respond to data subject requests within 30 days of receiving them. For complex requests, you may extend this by an additional 60 days with notification.

Document all DSR responses in your internal records, including: request date, nature of request, action taken, response sent date.
