⚡ Edge Functions & Automation

Edge functions reference

All backend logic in Community Carpool runs as Deno-based Supabase Edge Functions. There are 14 live deployed functions as of March 2026, plus 2 planned replacements in progress. Source files are in Supabse Edge Functions/ (note folder name typo — do not rename).

What verify_jwt means

Every edge function has a verify_jwt setting declared in its config.toml. This controls whether Supabase validates the caller's JWT before the function code runs:

  • verify_jwt: true — Supabase checks the Authorization: Bearer <token> header before invoking the function. Requests with a missing, invalid, or expired token are rejected with 401 before any function code runs. The token must be the current anon key (or a service-role key). Cron jobs calling verify_jwt: true functions work because Supabase internally passes a valid service-level JWT when invoking scheduled functions.
  • verify_jwt: false — No JWT check at the Supabase layer. The function is responsible for its own authentication — for example, validating a ?token= query parameter, checking a deletion token, or verifying a session token in the Authorization header using its own logic.
⚠️
verify_jwt: false does not mean the function is unauthenticated — it means auth is handled inside the function itself. Never assume a false function is publicly writable.

Live functions (14 deployed)

Function Trigger verify_jwt Description
submit-journey Frontend POST true Main journey submission handler. Pipeline: checks blacklist → creates or updates user record (stores terms_accepted_version, ref_code, UTM attribution) → enforces journey limit (from max_journeys_per_user config key, default 10) → resolves org_id from org_code → calculates road distance via Mapbox API (falls back to haversine if unavailable) → inserts submission → triggers find-matches if matching_mode is instant or hybrid.
find-matches Called by submit-journey false Spatial matching algorithm — not called directly by the frontend. Uses the find_nearby_users PostGIS RPC to find submissions within proximity thresholds. Computes match_strength score (0–100) from three weighted components: direction_weight, distance_weight, and timing_weight (all configurable). Creates match records in the matches table and returns match count.
get-matches-page Frontend GET ?token= true Returns all data for a user's matches page, identified by their journey token in the URL. Response includes the user's active journeys, all matches with status and scores, in-app notifications, and the poll_interval config value for auto-refresh. Checks required_terms_version and prompts for T&C re-acceptance if needed. Applies privacy masking — first names are partially masked and emails are hidden until mutual_confirmed status. Returns 401 if the token has expired.
update-match-status Frontend POST true Handles all user actions on a match. Supported actions: interest_expressed (one party is interested), not_interested (decline — hides the match), reset (undo interest/decline). Also handles accept_terms action for T&C re-acceptance flow. When both parties express interest, match status transitions to mutual_confirmed and a mutual match email is sent to both via Resend, revealing full names and email addresses.
update-email-prefs Frontend GET / POST true Manages user email preferences. GET returns the current preference values. POST updates them. Accepts camelCase body keys: unsubscribedMatches, unsubscribedReminders, unsubscribedMarketing. Preferences are stored in the user_notifications table.
manage-deletion Frontend POST false Two-step account deletion flow. Step 1: user submits their email — the function sends a deletion confirmation email with a time-limited token (unregistered emails silently succeed to prevent user enumeration). Step 2: user clicks the link in the email — the function validates the token and sets deletion_requested_at on the user record, scheduling the account for deletion by process-deletions. Planned to be split into two separate functions (request-deletion and confirm-deletion) in a future release.
submit-support-ticket Frontend POST false Creates a support ticket and stores it to the support_tickets table. Accepts three ticket types: technical, report_user, general_enquiry. Rejects type deletion with 400 — users must use the manage-deletion flow instead. Sends a notification email to the address in the support_notify_email config key.
batch-send-emails Cron (scheduled) true Sends match notification emails to users who have unsent matches. Respects two config flags: testing_mode (when true, only sends to users with email_whitelist = true) and match_notification_enabled (master kill switch — when false, no emails are sent at all). Updates sent matches from new to notified status.
process-deletions Cron (daily, 22:00 UTC) true Processes accounts where the deletion grace period has elapsed (deletion_requested_at + data_retention_days config days). Deletes in FK-safe order: events → matches → submissions → user_notifications → support_tickets → users. Inserts a record to deletion_log for each account deleted. Runs daily at 22:00 UTC (02:00 GMT+4).
expire-journeys Cron (scheduled) true Marks submissions whose expires_at timestamp has passed as expired. Expired submissions are excluded from matching. Runs on a scheduled basis to keep journey status current without requiring user action.
track-event Frontend POST false General-purpose event logging endpoint. Accepts event data (event type, optional user_id, submission_id, match_id, and a JSONB metadata payload) and writes to the events table. Used for analytics and behavioural tracking without requiring authenticated sessions.
admin-auth Admin panel false Handles all admin authentication flows: login (verifies password using PBKDF2 against the stored hash), logout (invalidates session), session validation, and forgot-password flow. On successful login, creates a session record in admin_sessions with a 24-hour expiry and returns the session token to the browser. Enforces is_active, role_expires_at, and allowed_ips checks on every login attempt.
admin-api Admin panel false Umbrella function for all admin dashboard operations. Uses its own session-token auth (reads token from Authorization header and validates against admin_sessions). Exposes endpoints for: platform metrics, support ticket management (view, add notes, reply, close), deletion queue management (view, cancel, process with PIN), user lookup, blacklist add/remove, audit log, team management (add/edit/deactivate admin users), and referral link management.
resend-webhook Resend HTTP callback false Receives email delivery event webhooks from Resend (e.g. delivered, bounced, opened, clicked) and stores them to the email_events table. Used for deliverability monitoring and diagnosing email issues. The webhook endpoint URL is registered in the Resend dashboard.

Engineering notes — bugs fixed

These bugs were identified and resolved. Documented here so they are not accidentally re-introduced:

PostGIS WKB vs GeoJSON

The Supabase JS client returns GEOGRAPHY columns as WKB hex strings, not GeoJSON. Accessing sub.from_point?.coordinates always returns undefined, causing lat/lng to resolve to 0,0 and producing no matches. Fix: Use the get_submission_coords(p_id) RPC function which returns float columns (from_lat, from_lng, to_lat, to_lng).

Module-level SES initialisation causing 502

Declaring const ses = new SESClient({...}) at the top level of a Deno edge function crashes the function at cold start if AWS secrets are missing, producing a 502 with deployment_id: null. Fix: Wrap in a lazy getter: function getSES() { return new SESClient({...}); } called only inside the request handler.

!inner + .or() returning empty results

Using !inner in a Supabase JS join combined with a .or('sub_a_id.eq.X,sub_b_id.eq.X') filter consistently returns an empty array. Fix: Reference FK column names directly in the join (e.g. submissions!sub_a_id) and remove !inner.
