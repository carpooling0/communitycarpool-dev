Managing team access

The admin panel uses its own role-based access control system, completely separate from Supabase Auth. Admin accounts live in the admin_users table. Sessions live in admin_sessions. Authentication is handled by the admin-auth edge function, and all panel operations go through admin-api. Only Super-Admins can add or modify admin accounts.

How admin sessions work

When an admin logs in successfully, admin-auth creates a session record in the admin_sessions table and returns the session token to the browser. The token is stored in localStorage and sent in the Authorization: Bearer <token> header on every subsequent request to admin-api.

  • Sessions expire after 24 hours from login. After expiry, the admin is redirected to the login page.
  • On every admin-api call, the session token is validated: it must exist in admin_sessions, not be expired, and belong to an account where is_active = true and role_expires_at (if set) has not passed.
  • Logging out invalidates the session record in the database immediately.

Role-based access control (RBAC)

There are three roles. Each role has a strict scope — a role cannot perform actions above its tier, regardless of what it requests from admin-api.

The deletion PIN

Any admin with the admin or super_admin role can trigger account deletions from the panel. Because deletion is irreversible, the panel requires a separate 6-digit PIN to confirm the action — in addition to the admin being logged in.

  • The PIN is stored as a hash in admin_users.deletion_pin using the same PBKDF2 method as the login password.
  • It is set when an admin account is created and can only be changed by a super_admin (or by the admin themselves if the panel provides a self-service change flow).
  • Every admin with admin or super_admin role must have a deletion PIN set — accounts without one cannot perform deletions.
  • The PIN is not the same as the login password — they are stored and verified separately.

To reset a deletion PIN via SQL:

-- Generate a PBKDF2 hash of the new PIN externally, then update:
UPDATE admin_users
SET deletion_pin = '[saltHex:hashHex — generated programmatically]'
WHERE email = 'admin@example.com';

Adding a new admin

New admin accounts can be created from the Team panel (super_admin only). To add an account directly via SQL (e.g. for the first super_admin seed):

INSERT INTO admin_users (email, name, role, password_hash, deletion_pin, is_active)
VALUES (
  'newadmin@example.com',
  'Admin Name',
  'admin',
  '[REDACTED — saltHex:hashHex, generated programmatically]',
  '[REDACTED — hashed 6-digit PIN]',
  true
);

Password hashing: PBKDF2 via Deno's crypto.subtle — 16-byte random salt, 100,000 iterations, SHA-256 digest, output as hex. Format: saltHex:hashHex. Use a small utility script or the admin-auth function's own hashing helper to generate these values.

Role expiry

The role_expires_at field auto-revokes access after a specified date. This is useful for contractors, seasonal support staff, or any temporary access arrangement.

UPDATE admin_users
SET role_expires_at = '2026-09-30 23:59:59+04'
WHERE email = 'contractor@example.com';

After the expiry timestamp passes, the admin's login is rejected with an "Access expired" error — even if is_active is still true. Set role_expires_at = NULL to remove the expiry and restore indefinite access.

IP allowlist

The allowed_ips field restricts an admin account to a list of specific IP addresses. Login attempts from any IP not in the list are rejected.

UPDATE admin_users
SET allowed_ips = ARRAY['203.0.113.10', '198.51.100.0']
WHERE email = 'admin@example.com';

Set allowed_ips = NULL (the default) to allow login from any IP. This field is most useful for machine accounts or admins who always access from a fixed office IP.

Deactivating an admin

Setting is_active = false immediately prevents all logins and rejects any in-flight session validation — the effect is instant.

UPDATE admin_users SET is_active = false WHERE email = 'leavingadmin@example.com';

The account remains in the database so that all historical admin_actions audit log entries remain attributable. Do not delete admin accounts unless legally required.

Seeding the first Super-Admin

The first super_admin account must be seeded directly in the database, since there is no panel to log into yet. Run the INSERT statement above with role = 'super_admin'. Once you can log in, all subsequent accounts can be managed from the Team panel.

Urgent access revocation

If you need to immediately cut off a team member's access (e.g. unexpected departure or security incident), the fastest path is:

  1. Set is_active = false via the Supabase SQL Editor or the admin panel Team page — this takes effect immediately for all subsequent API calls.
  2. Invalidate any open sessions by deleting them from admin_sessions:
    DELETE FROM admin_sessions
WHERE admin_id = (SELECT admin_id FROM admin_users WHERE email = 'leavingadmin@example.com');
  3. If they had access to any shared secrets (Resend API key, Supabase service role key, etc.), rotate those immediately — see Rotating secrets.

Offboarding checklist

When a team member leaves:

  • Set is_active = false on their admin account immediately
  • Delete their active sessions from admin_sessions
  • Rotate any shared secrets they had access to — see Rotating secrets
  • Remove their personal access tokens and SSH keys from GitHub (if applicable)
  • Update any shared service logins they could access (Resend dashboard, Google Cloud Console, domain registrar)
  • Review recent admin_actions and admin_audit_log entries for their account to confirm no unexpected actions were taken

Best practices

  • Never share admin credentials. Every team member must have their own account. Shared accounts make the audit log meaningless.
  • Always set role expiry for contractors and temporary staff. It is easier to extend access than to remember to revoke it.
  • Always set a deletion PIN for admin+ roles. Accounts without a PIN cannot process deletions — this is by design, not a gap to work around.
  • Use IP allowlists for high-privilege accounts where practical, especially for super_admin accounts that are rarely used from unknown networks.
  • Assign the least privileged role that meets the need. Support staff handling tickets do not need the admin role.
  • Review the audit log periodically. The admin_actions table is your primary accountability trail for anything that changes user data.
