Supabase dashboards

Supabase is the core backend for Community Carpool — it hosts the PostgreSQL database, edge functions, authentication, storage, and cron job scheduling. There are two separate Supabase projects: one for production and one for development.

⚠️
Credentials are redacted in these docs. All passwords, keys, and account logins are stored securely in the team password manager. Never store credentials in plain text or share them over email.

Key areas of the Supabase dashboard

Table Editor

Browse and edit database rows visually. Useful for quick lookups — e.g. checking a user's record or reading config values. For complex queries, use the SQL Editor instead.

SQL Editor

Run arbitrary SQL against the database. This is where you apply one-off queries, inspect data, and run manual migrations. All queries run as the service role (full access, RLS bypassed).

⚠️
No undo. Destructive SQL (DELETE, DROP, TRUNCATE) cannot be undone. Always download a backup before running bulk changes on production. See Backup & recovery.

Edge Functions

Lists all deployed edge functions with their status, last deployed timestamp, and region. From here you can view individual function logs, trigger scheduled functions manually, and manage environment variables (secrets).

Logs

Two log streams are most useful:

  • Edge Function Logs — all function invocations, console output, errors, and HTTP status codes. Filter by function name and time range.
  • Postgres Logs — DB-level query errors, slow queries, and connection issues.

Database > Cron Jobs

Supabase uses pg_cron for scheduled tasks. Cron jobs are listed here. You can also manage scheduled edge function calls from Edge Functions > Scheduled Functions. See Cron jobs & scheduled tasks for full details.

Auth settings

Community Carpool does not use Supabase Auth for end-users (users authenticate via secure token links, not accounts). This panel is used only for managing the Supabase service itself. The admin panel uses its own admin_users table and session management.

API settings

Located at Settings > API. Contains:

  • Project URL — the API endpoint used in frontend code and edge functions
  • Anon (public) key — safe to expose in frontend; RLS prevents direct table access by unauthenticated clients
  • Service role key — bypasses RLS; used only inside edge functions. Never expose this key in frontend code or commit it to git.
🚫
Service role key — handle with extreme care. This key grants unrestricted database access. It is stored only in Supabase Vault (as a secret accessible to edge functions) and in the team password manager. If this key is ever exposed, treat it as a security incident immediately — see Incident runbook.

Storage

Supabase Storage is not currently used by Community Carpool. No buckets are configured.

Row Level Security (RLS)

RLS is enabled on all application tables. However, no RLS policies are defined — this is intentional. The effect is:

  • Anon (unauthenticated) clients cannot read or write any table directly
  • Edge functions using the service role key bypass RLS and have full access
  • This means all data access goes through edge functions — direct table access from the frontend is blocked

The exception is spatial_ref_sys (a PostGIS system table) — RLS cannot be enabled on it. This is expected and safe to ignore in the Supabase security advisor warnings.

ℹ️
Tip: If the security advisor shows "RLS disabled" warnings for spatial_ref_sys, you can safely dismiss them. All other tables should have RLS enabled.

Running SQL queries

  1. Open the Supabase dashboard for the correct project (dev or prod)
  2. Click SQL Editor in the left sidebar
  3. Click New query
  4. Type your SQL and click Run (or press Ctrl+Enter)
  5. Results appear in the panel below
-- Example: read config table
SELECT key, value FROM config ORDER BY key;

-- Example: check a user by email
SELECT user_id, first_name, last_name, email, deletion_requested_at
FROM users WHERE email = 'user@example.com';

Viewing edge function logs

  1. Go to Logs in the left sidebar
  2. Select Edge Functions
  3. Use the function filter dropdown to narrow to a specific function
  4. Set the time range (default is last hour)
  5. Click any log entry to expand the full output including console.log lines
💡
When debugging a failing function, look for lines starting with Error or HTTP status 5xx. The Supabase log viewer also shows the request payload and response for each invocation.
