Monitoring & logs

Community Carpool does not have automated alerting configured. All monitoring is currently manual — periodic log review in the Supabase dashboard and Resend. This page documents what to monitor, where to find logs, and what to look for.

Log sources

Log sourceWhere to find itWhat it covers
Edge Function Logs Supabase Dashboard > Logs > Edge Functions All function invocations: HTTP status, response time, console.log output, errors
Postgres Logs Supabase Dashboard > Logs > Postgres DB query errors, slow queries, connection issues
Email delivery logs resend.com/emails Email sent, delivered, bounced, opened, clicked
Admin audit log admin_actions table in DB (planned — admin panel Audit Log panel) Every admin action: who did what, when, to which record
GitHub Actions github.com/carpooling0/communitycarpool/actions GitHub Pages deployment status — no CI/CD beyond Pages deploys

Edge Function Logs — what to look for

HTTP status codes

  • 200 — Success. Expected for normal operation.
  • 401 — Unauthorised. JWT missing or expired. Check anon key validity; check if a cron job lost its auth.
  • 400/404 — Bad request or not found. Usually user error (invalid token, missing field). Not necessarily a system problem.
  • 500/502 — Server error. Investigate immediately. Common causes: missing secret, bad deploy, module-level code crash at cold start.

Key things to check weekly

  • batch-send-emails — ran at the expected time? HTTP 200? How many emails sent?
  • process-deletions — ran at 22:00 UTC? Any deletions processed?
  • submit-journey — any unexpected 4xx/5xx spikes? Unusual volume?

Viewing logs

  1. Go to Supabase Dashboard > Logs > Edge Functions
  2. Select the function from the dropdown (or leave on "All functions" for overview)
  3. Set the time range (last 24h is a good daily health check)
  4. Click any log entry to expand and see the full response body and console output

Admin audit log

The admin_actions table records every action taken by admins. Query it to review recent activity:

-- Recent admin actions
SELECT
  aa.created_at,
  au.name as admin_name,
  aa.action_type,
  aa.target_type,
  aa.target_id,
  aa.notes
FROM admin_actions aa
JOIN admin_users au ON aa.admin_id = au.admin_id
ORDER BY aa.created_at DESC
LIMIT 50;

Resend email logs

  1. Go to resend.com/emails
  2. Review delivery status for recent sends
  3. Filter by date or recipient if investigating a specific complaint
  4. Bounced emails indicate invalid addresses — consider investigating if a user reports not receiving emails

Checking current usage by service

Check these dashboards monthly — or any time you suspect you're approaching a limit. Each service is on a free or low-volume tier; exceeding limits triggers either throttling or unexpected charges.

If you hit 3,000 emails/month you will need to upgrade to the Starter plan ($20/month for 50,000 emails). The free tier has a hard stop — emails over the limit are rejected, not queued.

GitHub Pages — bandwidth

What to checkWhere to find itFree limit

Builds remainingSame page — GitHub Pages has a limit of 10 builds/hour10 builds / hour

GitHub Pages bandwidth is essentially unlimited at current scale — the site is lightweight static HTML/CSS/JS with no large assets. You would need millions of page views per month to approach the 100 GB limit. The more likely constraint is repository size if large files (images, zips) are ever committed accidentally.

💡
GitHub does not provide a self-serve bandwidth dashboard for Pages on free accounts. If you want detailed traffic analytics, add a free tool like Plausible or Google Analytics to track page views and sessions.

Setting up alerting (recommended)

No automated alerting is currently configured. Recommended future setup:

  • Supabase: Set up a Supabase Alert (in Settings > Alerts) to notify on high error rates or high database CPU
  • Uptime monitoring: Use a free tool like UptimeRobot to monitor communitycarpool.org and alert on downtime
  • Resend: Monitor bounce rate — high bounce rates can get your domain flagged as spam
  • Google Cloud: Set billing alerts (see Cost & billing)
💡
A simple weekly log review (5–10 minutes) is the minimum recommended monitoring practice. Check: cron jobs ran, no 5xx errors, email deliveries looking healthy.
