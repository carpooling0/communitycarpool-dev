🚨 Compliance & Incidents

Incident runbook

Step-by-step response procedures for the most common operational incidents. Use these as a starting point — document what you did and the outcome for each incident so the runbook can be improved over time.

🔴 Scenario 1: Edge function returning 500 / 502

Symptoms: Users reporting errors submitting journeys, the site returning error messages, or cron jobs failing silently.

  1. Go to Supabase Logs > Edge Functions. Filter by the affected function. Look for 500 or 502 status codes.
  2. Click the error log entry and read the error message in full. Common causes:
    • Missing secret — Cannot read property of undefined or authentication errors from Resend/external APIs. Run supabase secrets list to verify all required secrets are set.
    • Bad deploy — syntax error or import failure in the function code. Redeploy from a known-good version.
    • Module-level init error — code at the top level of the function file fails during cold start. Move initialisation inside the request handler.
  3. Fix the underlying issue, then redeploy: 
    supabase link --project-ref tbkjealpnoriwdosvmju
supabase functions deploy [function-name]
  4. Test by invoking the function manually from the dashboard or submitting a test journey.
  5. Monitor logs for the next 10 minutes to confirm the fix.
🟡 Scenario 2: No match emails being sent

Symptoms: Users with matches are not receiving notification emails. No email delivery records in Resend.

  1. Check testing_mode: SELECT value FROM config WHERE key = 'testing_mode'. If true, emails only go to whitelisted users — this is expected behaviour, not a bug.
  2. Check match_notification_enabled: SELECT value FROM config WHERE key = 'match_notification_enabled'. If false, email sending is disabled.
  3. Check if the cron job is running. Go to Supabase Logs > Edge Functions, filter by batch-send-emails, and look for recent entries at the scheduled time (22:00 UTC). If missing, the cron job may have been lost — re-add it via the Supabase dashboard.
  4. Check RESEND_API_KEY is set: supabase secrets list. If missing, set it and redeploy.
  5. Trigger batch-send-emails manually from Edge Functions dashboard and watch the logs.
  6. Check the Resend dashboard for delivery failures or rate limiting.
🟡 Scenario 3: User can't access matches page (401 or "link expired")

Symptoms: User follows the link from their email and sees an error or access denied page.

  1. Check if the token has expired. The expiry is set to created_at + match_token_expiry_days (currently 120 days): 
    SELECT journey_token_expires_at, last_seen_at FROM users WHERE email = 'user@example.com';
  2. If expired: generate a new token and update the user's record. Send them a new matches link via the support email.
  3. If not expired: check whether the anon key embedded in the frontend is current. An outdated anon key in config.js will cause all JWT-verified requests to fail with 401.
  4. Check match_token_expiry_days config value — if it was recently changed to a lower value, some existing tokens may now be considered expired retroactively.
🟡 Scenario 4: Bulk data issue — wrong matches or no matches created

Symptoms: Users submitting journeys are getting no matches despite others being in similar areas. Or matches seem incorrect.

  1. Do not delete anything. Investigate first — use the Admin panel or SQL to look at the matches table.
  2. Check that get_submission_coords RPC exists and returns float coordinates: 
    SELECT * FROM get_submission_coords(1); -- replace 1 with a real sub_id
    If this returns an error, the RPC function was lost (e.g. after a restore). Re-apply the migration.
  3. Check the find_nearby_users function returns from_lat, from_lng, to_lat, to_lng as float columns (not WKB hex). If it returns hex strings, re-apply the updated function migration.
  4. Check match_strength values in recent matches: SELECT match_strength, COUNT(*) FROM matches GROUP BY match_strength ORDER BY match_strength. All 0s or all 100s would indicate a data problem.
  5. If find-matches ran with bad coordinate data (all 0,0), existing matches may have near-zero match strength. These can be re-evaluated by re-running find-matches for affected submissions — but do not delete matches with existing user interactions (interest expressed, mutual confirmed).
🔴 Scenario 5: Unauthorised access — suspected security breach

Symptoms: Suspicious admin activity in audit log, users reporting privacy violations, or evidence that credentials may have been exposed.

  1. Deactivate the suspected admin account immediately (if breach is through admin panel): 
    UPDATE admin_users SET is_active = false WHERE email = 'suspected@account.com';
  2. Review the admin_actions audit log for the account: 
    SELECT * FROM admin_actions WHERE admin_id = [id] ORDER BY created_at DESC LIMIT 100;
  3. If the service role key may have been exposed:
    • Contact Supabase support immediately at supabase.com/support — request emergency key rotation
    • Until rotated, all data must be considered potentially accessible
  4. If Resend API key may have been exposed: rotate it immediately (see Rotating secrets).
  5. If Google Maps key may have been exposed: restrict it in Google Cloud Console immediately, then generate a new key.
  6. Audit recent data access — look for unusual query patterns or data exports in Supabase Postgres logs.
  7. Notify affected users if personal data was accessed by an unauthorised party. Consult applicable data protection regulations for notification timelines.
  8. Document the incident fully: timeline, what was accessed, what action was taken, who was notified.
🟡 Scenario 6: Site down (GitHub Pages outage or DNS failure)

Symptoms: communitycarpool.org is unreachable. DNS lookup fails or site returns an error.

  1. Check GitHub status at githubstatus.com — if GitHub Pages is degraded, wait for recovery.
  2. Check DNS by running: dig communitycarpool.org A — should return the GitHub Pages IPs (185.199.108-111.153).
  3. If DNS is wrong: log into the domain registrar and verify the A records match the GitHub Pages IPs listed in DNS records.
  4. If GitHub Pages reports the site is up but the custom domain is not working: check the CNAME file in the repo root still contains communitycarpool.org.
  5. SSL issues (expired/invalid cert): GitHub Pages auto-renews Let's Encrypt certificates. If expired, wait — usually resolves within 24h. Contact GitHub support if it persists beyond 48h.
💡
After any incident: write a short post-mortem. What happened, why, how it was fixed, and what can be done to prevent it. Even a few bullet points helps. Store these in your team's documentation or communication channel.
