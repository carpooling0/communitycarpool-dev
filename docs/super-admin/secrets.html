Secrets & API keys

Secrets are sensitive values (API keys, tokens, passwords) that edge functions need at runtime. They are stored in the Supabase Vault — encrypted at rest and never exposed in logs or dashboard UI. This page documents what each secret is, where it comes from, and where it lives.

🚫
Never commit secrets to git, paste them in chat messages, or include them in documentation. All actual secret values are in the team password manager only. This page documents secret names and their purpose — not their values.

Supabase Vault secrets (edge function environment variables)

Secret name Status Purpose Where to get it
RESEND_API_KEY Active Authenticates calls to the Resend email API. Used in batch-send-emails and manage-deletion. resend.com > API Keys > Create API Key
SUPABASE_SERVICE_ROLE_KEY Active Allows edge functions to bypass RLS and access all tables. Auto-available in edge functions — you typically don't need to set this manually. Supabase Dashboard > Settings > API > service_role key
SUPABASE_URL Active The project API URL. Auto-available in edge functions. Supabase Dashboard > Settings > API > Project URL
AWS_REGION Future AWS region for SES email sending (e.g. us-east-1). AWS Console — the region where SES is configured
AWS_ACCESS_KEY_ID Future AWS IAM access key for SES authentication. AWS Console > IAM > Users > Security credentials
AWS_SECRET_ACCESS_KEY Future AWS IAM secret key paired with the access key ID. AWS Console > IAM > Users > Security credentials (shown only once at creation)
SES_FROM_EMAIL Future Verified sender email address for AWS SES (e.g. noreply@communitycarpool.org). AWS SES > Verified identities

Frontend credentials (in config.js / HTML)

These are technically accessible in the browser, but are safe because of their nature or additional restrictions applied:

Credential Location Safe because
Supabase anon key config.js (committed to git) Designed to be public. RLS blocks all direct table access from the anon role — no policies = anon has zero access. Only works to call edge functions.
Google Maps API key index.html script tag Restricted by HTTP referrer to *.communitycarpool.org/* in Google Cloud Console. Useless from other origins.
ℹ️
The Supabase anon key is a JWT that identifies the project and role, not a password. It cannot be used to access tables directly when RLS is configured correctly (which it is). It is safe to include in public frontend code.

Setting secrets via CLI

# Set a single secret (link to correct project first)
supabase secrets set RESEND_API_KEY=re_xxxxxxxxxxxxxxxxxxxxxxxxxx

# Set multiple secrets from a .env file
supabase secrets set --env-file .env.production

# List current secrets (names only — values never shown)
supabase secrets list

Setting secrets via dashboard

  1. Go to Supabase Dashboard > Edge Functions
  2. Click on any function
  3. Go to the Secrets tab (or Settings)
  4. Add/update secrets there

Note: secrets set via the dashboard apply to all edge functions in the project.

What happens if a secret is missing

If an edge function calls Deno.env.get('RESEND_API_KEY') and the secret is not set, it returns undefined. Depending on how the function uses it, this can cause:

  • A 401 from Resend (if undefined is passed as the Bearer token)
  • A JavaScript error if the value is used without null-checking
  • A 502 at cold start if the secret is used in module-level initialisation (see known bugs)

If you see unexpected errors after deploying or after a secret rotation, check that all required secrets are set via supabase secrets list.
