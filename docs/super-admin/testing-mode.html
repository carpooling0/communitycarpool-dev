Testing mode

Testing mode is a safety guard that prevents real emails from being sent to all users during development and early launch. When enabled, match notification emails are only sent to explicitly whitelisted users.

Current state

Config keyCurrent valueEffect
testing_modetrueEmails only go to users with email_whitelist = true
match_notification_enabledtrueEmail sending is enabled (but gated by testing_mode)

As of the last recorded state: 68 matches pending notification, 0 users whitelisted. Emails will not be sent until at least one user is whitelisted or testing_mode is disabled.

What testing mode affects

FeatureAffected by testing_mode?Notes
batch-send-emailsYesSkips users without email_whitelist = true when testing_mode is true
Deletion confirmation emailsNoAlways sent regardless — users need these to complete deletion
submit-journeyNoJourneys are created normally
find-matchesNoMatching runs normally — matches are created, just not emailed
get-matches-pageNoUsers can still view their matches page if they have the link

Whitelisting a user for testing

To receive test emails while testing_mode is active, add email_whitelist = true to a specific user:

UPDATE users SET email_whitelist = true WHERE email = 'your@email.com';

That user will then receive match notification emails on the next batch-send-emails cron run. To trigger immediately, invoke the function manually from the Supabase dashboard.

Removing a whitelist entry

UPDATE users SET email_whitelist = false WHERE email = 'your@email.com';

Disabling testing mode (going live)

⚠️
This will send real emails to all users with pending match notifications on the next batch-send-emails cron run. As of the last recorded state, that is 68 matches. Confirm you are ready before proceeding.

Pre-launch checklist

  • Resend domain is verified (SPF + DKIM showing green)
  • RESEND_API_KEY secret is set in Supabase
  • Match notification email template looks correct (test with a whitelisted user first)
  • Matches page link in the email goes to the correct URL
  • Resend account is on a plan that supports the expected email volume
  • All team members are aware of the go-live

Disable testing mode

UPDATE config SET value = 'false' WHERE key = 'testing_mode';

The next scheduled run of batch-send-emails will email all users with pending notifications. To control timing, disable testing mode just before the cron job runs (e.g. at 21:50 UTC for a 22:00 cron), or trigger the function manually immediately after disabling.

Re-enabling testing mode

UPDATE config SET value = 'true' WHERE key = 'testing_mode';

This is safe to do at any time. It won't undo emails already sent — it only prevents future batch emails from going to non-whitelisted users.

Pausing all email sending

If you need to stop all email sending immediately (e.g. a Resend issue or accidental misconfiguration), use the master switch:

UPDATE config SET value = 'false' WHERE key = 'match_notification_enabled';

This stops batch-send-emails from sending any emails regardless of testing_mode or whitelist status. Useful as an emergency brake without touching testing_mode.
