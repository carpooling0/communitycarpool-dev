Database migrations

Database migrations are SQL scripts that modify the schema — adding tables, columns, indexes, or functions. All schema changes must be applied as migrations to keep development and production in sync.

⚠️
Always back up before migrating production. Download a backup from Supabase Dashboard > Settings > Backups before running any schema change on production. See Backup & recovery.

Where migrations live

Migration files are stored in supabase/migrations/ in the repository root. Each migration is a .sql file named with a timestamp prefix:

supabase/migrations/
  20260101120000_initial_schema.sql
  20260218000000_add_interest_columns_to_matches.sql
  20260218000001_get_submission_coords_rpc.sql
  ...

Naming convention

Migration filenames follow the format: YYYYMMDDHHMMSS_description.sql

  • Timestamp ensures migrations run in chronological order
  • Description should be brief and lowercase with underscores
  • Example: 20260315090000_add_email_whitelist_column.sql

Applying a migration

Option 1: Supabase SQL Editor (recommended for quick changes)

  1. Open the Supabase dashboard for the correct project
  2. Go to SQL Editor
  3. Paste the migration SQL
  4. Click Run
  5. Save the SQL as a migration file in supabase/migrations/ for future reference

Option 2: Supabase CLI (recommended for repeatable deployments)

# Ensure the CLI is linked to the correct project
supabase link --project-ref tbkjealpnoriwdosvmju

# Push all unapplied migrations
supabase db push

The CLI tracks which migrations have been applied using an internal table. Only unapplied migrations are run.

Writing safe migrations

Follow these principles to avoid locking tables or causing downtime:

Always use IF NOT EXISTS / IF EXISTS guards

-- Safe: won't fail if column already exists
ALTER TABLE users ADD COLUMN IF NOT EXISTS email_whitelist BOOLEAN DEFAULT false;

-- Safe: won't fail if table already exists
CREATE TABLE IF NOT EXISTS blacklist (
  email TEXT PRIMARY KEY,
  added_at TIMESTAMPTZ DEFAULT now()
);

Avoid dropping columns on production

Instead, mark columns as deprecated and remove them in a later migration once confirmed they are no longer referenced in any edge function code.

Never run without a WHERE clause on large tables

-- DANGEROUS — deletes all rows
DELETE FROM submissions;

-- Safe — targeted delete
DELETE FROM submissions WHERE expires_at < now() AND journey_status = 'expired';

Test on dev first

Apply every migration to the dev Supabase project first. Verify the schema change works correctly before promoting to production.

Known applied migrations (key ones)

Migration nameDescription
initial_schemaCreates all base tables: users, submissions, matches, organisations, support_tickets, deletion_log, config, blacklist
add_interest_columns_to_matchesAdds interest_a, interest_b, interest_a_at, interest_b_at columns to matches
get_submission_coords_rpcCreates the get_submission_coords(p_id INT) RPC function that returns lat/lng as floats (not PostGIS WKB)
find_nearby_users_updateUpdates find_nearby_users() RPC to return from_lat, from_lng, to_lat, to_lng as FLOAT columns

For the complete list of applied migrations, check Supabase Dashboard > Database > Migrations.

RPC functions

In addition to table migrations, custom PostgreSQL functions are used to work around Supabase JS client limitations. Key functions:

  • get_submission_coords(p_id INT) — returns a submission with lat/lng as floats rather than PostGIS WKB hex strings. Used by find-matches.
  • find_nearby_users(...) — spatial matching query that returns potential matches with float coordinate columns. Called by find-matches.

These functions are defined in the migration files and can be viewed/edited in Supabase Dashboard > Database > Functions.
