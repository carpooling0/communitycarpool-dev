Blacklist & moderation

The blacklist is a simple table of email addresses that are blocked from submitting journeys. It is the primary moderation tool — use it when a user has been reported for inappropriate behaviour or when an account is identified as abusive.

How the blacklist works

When a user submits a journey, submit-journey checks their email against the blacklist table. If a match is found, the submission is rejected with a 403 response. The user sees an error message but is not told why they were blocked.

Being on the blacklist does not delete the user's existing data. To fully remove them, use the admin deletion flow as well.

Current state

As of last recorded state: 2 emails on the blacklist. Check the current count:

SELECT COUNT(*) FROM blacklist;
SELECT * FROM blacklist ORDER BY added_at DESC;

Adding an email to the blacklist

INSERT INTO blacklist (email, added_by, reason)
VALUES (
  'badactor@example.com',
  'Admin Name',
  'report_user ticket #47 — inappropriate contact with another user'
);

The added_at column defaults to now(). Record the reason clearly — it helps future admins understand why a block was placed.

Removing an email from the blacklist

-- Remove a specific email
DELETE FROM blacklist WHERE email = 'user@example.com';
⚠️
Before removing a blacklist entry, review the original reason for the block. If the original report is still valid, do not remove it. Document the decision in admin notes on the related support ticket.

Reviewing reported users

When a report_user support ticket is received:

  1. Search for the reported email in the users table: SELECT * FROM users WHERE email = 'reported@example.com'
  2. Check their submission history: SELECT * FROM submissions WHERE user_id = [id]
  3. Review the ticket message for specific claims
  4. If the report is credible: add to blacklist AND consider processing an admin-initiated deletion
  5. Close the support ticket with admin notes documenting the decision

Moderation principles

  • The blacklist blocks future submissions but does not auto-delete existing data — handle that separately if needed
  • Always document the reason for blacklisting — this creates an audit trail
  • A user on the blacklist can still access their matches page if they have the link (the blacklist only blocks new submissions)
  • If a user's existing matches should be cancelled, manually update their match statuses to user_deleted
💡
For clearly abusive accounts, process both steps: add to blacklist AND initiate an admin deletion. This ensures their data is removed after the grace period and they cannot resubmit.
