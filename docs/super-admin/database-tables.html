🗄️ Database

Tables reference

Complete schema reference for all 20 database tables. RLS is enabled on all application tables with no policies defined — this means the anon role has zero direct table access, and all data operations go through edge functions using the service role key. The PostGIS system table spatial_ref_sys cannot have RLS enabled; this is expected and safe.

ℹ️
All tables listed here are live in the production database. There are no planned-only tables remaining.
Core journey data
users
One record per registered user. Created (or updated) when a user submits their first journey via submit-journey. The journey_token is the user's sole identifier for accessing their matches page.
ColumnTypeDescription
user_idSERIAL PKAuto-incrementing primary key
emailTEXT UNIQUE NOT NULLUser's email address — used as the unique identifier across all flows
first_nameTEXTFirst name as entered on the form
last_nameTEXTLast name as entered on the form
org_codeTEXTOrganisation code at the time of registration — links user to a record in organisations
journey_tokenTEXTSecure random token embedded in the matches page URL (/matches.html?token=...)
journey_token_expires_atTIMESTAMPTZToken expiry — set to created_at + match_token_expiry_days config value (currently 120 days)
last_seen_atTIMESTAMPTZLast time the user successfully loaded their matches page
deletion_requested_atTIMESTAMPTZWhen the user confirmed a deletion request — null if no pending deletion
deletion_tokenTEXTOne-time token sent in the deletion confirmation email
deletion_token_expires_atTIMESTAMPTZExpiry for the deletion confirmation token (typically 24 hours from issuance)
terms_accepted_versionTEXTVersion string of the T&C the user last accepted (e.g. "1.1")
terms_accepted_atTIMESTAMPTZTimestamp when the user last accepted the T&C
email_whitelistBOOLEAN DEFAULT falseWhen true, user receives emails even when testing_mode = true
ref_codeTEXTReferral code used at registration — matches a code in referral_links
created_atTIMESTAMPTZ DEFAULT now()Account creation timestamp
submissions
Each row is a journey submission. One user can have multiple submissions (e.g. different routes or schedules). Submissions are matched against each other by find-matches. Submissions expire after the configured number of days and are marked expired by the expire-journeys cron.
ColumnTypeDescription
sub_idSERIAL PKAuto-incrementing primary key
user_idINT FK → usersForeign key to the submitting user
from_locationTEXTHuman-readable home/start address (as returned by Google Maps autocomplete)
to_locationTEXTHuman-readable work/destination address
from_pointGEOGRAPHY(Point)PostGIS geography point for the start location. Stored as WKB internally — use the get_submission_coords RPC to retrieve as float lat/lng values.
to_pointGEOGRAPHY(Point)PostGIS geography point for the destination
distance_prefTEXTHow far the user is willing to travel to meet a carpool partner (e.g. "500m", "1km", "2km")
work_scheduleTEXTDays/times the user commutes (e.g. "Mon-Fri", "flexible")
org_idINT FK → organisationsResolved organisation ID at submission time (looked up from org_code)
org_codeTEXTOrganisation code as submitted on the form
countryTEXTCountry of the submission (set from form, informational)
journey_statusTEXTactive | expired | archived | deletion_pending
created_atTIMESTAMPTZ DEFAULT now()Submission timestamp
expires_atTIMESTAMPTZSet to created_at + configured expiry days. Submission excluded from matching after this date.
terms_versionTEXTVersion of T&C accepted at time of submission
matches
Each row represents a potential carpool pair between two submissions. Created by find-matches. The status field tracks the full match lifecycle from creation through to mutual confirmation or decline.
ColumnTypeDescription
match_idSERIAL PKAuto-incrementing primary key
sub_a_idINT FK → submissionsFirst submission in the pair
sub_b_idINT FK → submissionsSecond submission in the pair
match_strengthINT (0–100)Compatibility score — higher is better. Computed from direction, distance, and timing weights configured in the config table.
statusTEXTnew | notified | viewed | interest_expressed | not_interested | mutual_confirmed | user_deleted
interest_aBOOLEANWhether user A has expressed interest in this match
interest_bBOOLEANWhether user B has expressed interest in this match
interest_a_atTIMESTAMPTZWhen user A expressed interest
interest_b_atTIMESTAMPTZWhen user B expressed interest
notified_atTIMESTAMPTZWhen the match notification email was sent to both users
created_atTIMESTAMPTZ DEFAULT now()When the match record was created by find-matches
Organisations
organisations
Partner organisations that use Community Carpool. Users and submissions can be linked to an org via the org_code field. Managed from the admin panel Organisations section.
ColumnTypeDescription
org_idSERIAL PKAuto-incrementing primary key
org_codeTEXT UNIQUE NOT NULLShort identifier used in URLs and form fields (e.g. ACME). Case-insensitive by convention.
nameTEXTFull organisation name (e.g. "Acme Corporation")
domainTEXTEmail domain for the organisation (e.g. acme.com) — reserved for future auto-detection
is_activeBOOLEAN DEFAULT trueWhen false, org does not appear in dropdowns and is excluded from matching logic
created_atTIMESTAMPTZ DEFAULT now()Organisation creation timestamp
org_locations
Named locations associated with an organisation — e.g. office addresses or carpool meeting points. Used to provide pre-filled destination options for org users on the journey form.
ColumnTypeDescription
location_idSERIAL PKAuto-incrementing primary key
org_idINT FK → organisationsThe organisation this location belongs to
location_nameTEXTHuman-readable name for the location (e.g. "HQ — DIFC Gate Building")
location_latlngTEXTCoordinates stored as a plain text "lat,lng" string (e.g. "25.2048,55.2708")
is_activeBOOLEAN DEFAULT trueWhen false, location is hidden from the form
created_atTIMESTAMPTZ DEFAULT now()Creation timestamp
User preferences & notifications
user_notifications
Per-user email preference record. One row per user. Managed via the update-email-prefs edge function (called from the matches page and unsubscribe link). Checked by batch-send-emails before sending any email.
ColumnTypeDescription
user_idINT FK → users PKOne row per user — primary key and FK to users
unsubscribed_matchesBOOLEAN DEFAULT falseWhen true, user does not receive match notification emails
unsubscribed_remindersBOOLEAN DEFAULT falseWhen true, user does not receive reminder emails
unsubscribed_marketingBOOLEAN DEFAULT falseWhen true, user does not receive marketing emails
updated_atTIMESTAMPTZLast time any preference was updated
user_attribution
Stores UTM and referral attribution data captured at the time of a user's first journey submission. One row per user. Used for campaign performance tracking.
ColumnTypeDescription
user_idINT FK → users PKOne row per user — primary key and FK to users
utm_sourceTEXTUTM source parameter (e.g. "email", "linkedin")
utm_mediumTEXTUTM medium parameter (e.g. "cpc", "organic")
utm_campaignTEXTUTM campaign name (e.g. "q1-launch")
created_atTIMESTAMPTZ DEFAULT now()Timestamp of attribution capture (same as first submission)
Support & moderation
support_tickets
Support requests submitted via the support page. Ticket type deletion is blocked at the edge function — users are directed to the dedicated deletion flow. Admins manage tickets from the admin panel.
ColumnTypeDescription
ticket_idSERIAL PKAuto-incrementing primary key
emailTEXT NOT NULLSubmitter's email address
typeTEXTreport_user | technical | general_enquiry
messageTEXTThe ticket body text submitted by the user
statusTEXT DEFAULT 'open'open | closed
admin_notesTEXTInternal notes added by admins — append-only, timestamped entries
closed_atTIMESTAMPTZWhen the ticket was closed
created_atTIMESTAMPTZ DEFAULT now()Submission timestamp
blacklist
Email addresses blocked from submitting journeys. Checked at the start of every submit-journey call. Managed from the admin panel Blacklist section.
ColumnTypeDescription
emailTEXT PRIMARY KEYBlacklisted email address
added_atTIMESTAMPTZ DEFAULT now()When the address was blacklisted
added_byTEXTWho added the entry — admin display name or "system"
reasonTEXTReason for blacklisting (e.g. "report_user ticket #42")
feedback
User feedback submissions collected from within the platform (e.g. post-match surveys or general product feedback forms).
ColumnTypeDescription
idSERIAL PKAuto-incrementing primary key
user_idINT FK → usersThe user who submitted feedback (nullable for anonymous feedback)
feedback_textTEXTThe feedback content
ratingINTOptional numeric rating
contextTEXTWhere feedback was collected from (e.g. "post_match", "general")
created_atTIMESTAMPTZ DEFAULT now()Submission timestamp
Campaigns & attribution
referral_links
Campaign and referral links used for user acquisition tracking. Each link has a unique ref_code that is captured in users.ref_code at registration. Managed from the admin panel Campaign section.
ColumnTypeDescription
idSERIAL PKAuto-incrementing primary key
modeTEXTLink mode/type (e.g. "campaign", "partner", "referral")
displayTEXTHuman-readable label shown in the admin panel
urlTEXTThe full referral URL including the ref_code parameter
ref_codeTEXT UNIQUE NOT NULLThe short code appended to URLs (e.g. "Q1-LAUNCH")
created_by_idINT FK → admin_usersAdmin who created the link
created_by_nameTEXTDisplay name of the creating admin at time of creation (denormalised for audit clarity)
created_atTIMESTAMPTZ DEFAULT now()Creation timestamp
Deletions & compliance
deletion_log
Immutable audit record of completed account deletions. Written by process-deletions. Contains metadata only — no personal content (messages, locations, etc.) is retained after a deletion is processed.
ColumnTypeDescription
log_idSERIAL PKAuto-incrementing primary key
user_idINTThe user_id of the deleted account (retained for reference; the users row no longer exists)
emailTEXTEmail of the deleted account
deleted_atTIMESTAMPTZWhen the deletion was processed by the cron
reasonTEXTReason for deletion (e.g. "user_requested", "admin_action")
submissions_deletedINTCount of submission records deleted for this user
matches_deletedINTCount of match records deleted for this user
Events & analytics
events
General-purpose event log written by the track-event edge function. Captures user-facing actions and platform events for analytics and debugging. Event type definitions live in the event_types table.
ColumnTypeDescription
idSERIAL PKAuto-incrementing primary key
event_typeTEXT FK → event_typesThe type of event (e.g. "match_viewed", "journey_submitted")
user_idINT FK → usersUser who triggered the event (nullable for anonymous events)
submission_idINT FK → submissionsRelated submission, if applicable
match_idINT FK → matchesRelated match, if applicable
metadataJSONBArbitrary event payload — additional context specific to the event type
created_atTIMESTAMPTZ DEFAULT now()Event timestamp
event_types
Registry of valid event type strings. Acts as a reference table to constrain the event_type values in the events table and provide human-readable descriptions.
ColumnTypeDescription
event_typeTEXT PRIMARY KEYEvent type identifier (snake_case string)
descriptionTEXTHuman-readable description of when this event is fired
created_atTIMESTAMPTZ DEFAULT now()When this event type was registered
email_events
Stores email delivery event webhooks received from Resend. Written by the resend-webhook edge function. Used for deliverability monitoring and diagnosing bounce or failure issues.
ColumnTypeDescription
idSERIAL PKAuto-incrementing primary key
resend_email_idTEXTResend's internal email ID — links the event to a specific send
event_typeTEXTResend event type (e.g. "email.delivered", "email.bounced", "email.opened", "email.clicked")
recipientTEXTRecipient email address
payloadJSONBFull raw webhook payload from Resend for debugging
created_atTIMESTAMPTZ DEFAULT now()Timestamp when the webhook was received
Configuration
config
Key-value store for all platform configuration. Edge functions read from this table on every request. Values are always stored as text and cast as needed. See Config table for a full list of keys and their meanings.
ColumnTypeDescription
keyTEXT PRIMARY KEYConfiguration key name (snake_case)
valueTEXT NOT NULLConfiguration value — always stored as text; cast in code as needed (e.g. to boolean, integer)
Admin panel
admin_users
Admin panel accounts. One row per team member with panel access. All authentication, RBAC, and session management is handled by the admin-auth and admin-api edge functions — the Supabase Auth system is not used for admin accounts.
ColumnTypeDescription
admin_idSERIAL PKAuto-incrementing primary key
emailTEXT UNIQUE NOT NULLAdmin email address — used as login identifier
nameTEXTDisplay name shown in the panel and audit log
roleTEXTsuper_admin | admin | support
password_hashTEXTPBKDF2 hash in format saltHex:hashHex (16-byte random salt, 100,000 iterations, SHA-256)
deletion_pinTEXTHashed 6-digit PIN required to confirm irreversible deletion actions. Stored using the same hashing method as the password.
is_activeBOOLEAN DEFAULT trueWhen false, all login attempts are immediately rejected. Set to false to revoke access without deleting the account.
role_expires_atTIMESTAMPTZOptional: if set, login fails after this date. Used for temporary/contractor access.
allowed_ipsTEXT[]Optional IP allowlist — if non-null, login is rejected from any IP not in the array. Leave null to allow login from any IP.
created_atTIMESTAMPTZ DEFAULT now()Account creation timestamp
admin_sessions
Active admin login sessions. A session record is created on each successful login and stored in the browser's localStorage by the admin panel. Every admin-api request validates the session token against this table. Sessions expire after 24 hours.
ColumnTypeDescription
session_idTEXT PKCryptographically random session token — sent in the Authorization header on every admin API request
admin_idINT FK → admin_usersThe logged-in admin
created_atTIMESTAMPTZLogin timestamp
expires_atTIMESTAMPTZSession expiry — 24 hours from creation
ip_addressTEXTIP address of the session at login
admin_actions
Immutable high-level audit log of all admin panel actions. Every significant action (blacklisting a user, cancelling a deletion, closing a ticket, etc.) is recorded here. Viewable in the admin panel Audit section (super_admin only).
ColumnTypeDescription
action_idSERIAL PKAuto-incrementing primary key
admin_idINT FK → admin_usersWhich admin performed the action
action_typeTEXTAction identifier (e.g. delete_user, close_ticket, add_blacklist, cancel_deletion)
target_typeTEXTEntity type affected (e.g. user, ticket, blacklist)
target_idTEXTID of the affected entity
notesTEXTOptional additional context or justification entered by the admin
created_atTIMESTAMPTZ DEFAULT now()Timestamp of the action
admin_audit_log
Lower-level system event log, separate from admin_actions. Captures system-level events such as login attempts (successful and failed), session creation/expiry, and automated processes. Complements admin_actions which records deliberate human actions.
ColumnTypeDescription
idSERIAL PKAuto-incrementing primary key
event_typeTEXTSystem event type (e.g. "login_success", "login_failed", "session_expired")
admin_idINT FK → admin_usersRelated admin account — nullable for events with no associated account (e.g. unknown email login attempt)
ip_addressTEXTIP address associated with the event
metadataJSONBAdditional event context (e.g. failure reason, user agent)
created_atTIMESTAMPTZ DEFAULT now()Event timestamp
