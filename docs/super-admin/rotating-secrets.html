Rotating secrets

Secret rotation is the process of replacing an old API key or credential with a new one. Rotate secrets annually, whenever a team member with access leaves, or immediately if a key may have been exposed.

When to rotate

  • Annual rotation — good security hygiene, even if no breach is suspected
  • Team member offboarding — anyone who had access to secrets should trigger rotation
  • Suspected exposure — if a secret may have been seen by an unauthorised party (e.g. accidentally committed to git, pasted in a chat), rotate immediately
  • After a security incident — rotate all secrets as part of incident response

Rotating the Resend API key

1

Generate a new key in Resend

Go to resend.com > API Keys > Create API Key. Give it a name like "communitycarpool-prod-2026". Copy the key immediately — it is shown only once.

2

Update in Supabase

supabase link --project-ref tbkjealpnoriwdosvmju
supabase secrets set RESEND_API_KEY=re_new_key_here
3

Test

Trigger batch-send-emails manually in the Supabase dashboard to verify emails are still sending correctly. Check the Resend dashboard for a new delivery record.

4

Revoke the old key

In the Resend dashboard, delete the old API key. Update the team password manager with the new key.

Rotating the Google Maps API key

1

Create a new key in Google Cloud Console

Go to APIs & Services > Credentials > Create Credentials > API key. Immediately apply referrer restrictions (see Google Maps & Mapbox).

2

Update in frontend

Replace the old key in config.js (or wherever it is stored in the frontend HTML). Commit the change.

3

Deploy to dev, test, then push to prod

Test autocomplete works on the dev site. Push to production once confirmed.

4

Delete the old key

In Google Cloud Console, delete the old API key after the new one is confirmed working. Update the password manager.

Supabase service role key

The service role key is not directly rotatable by end users — it is managed by Supabase and tied to the project. If you believe this key has been compromised:

  1. Contact Supabase support immediately via supabase.com/support
  2. Request an emergency key rotation
  3. After rotation, update the key in the team password manager
  4. Update any local .env files and redeploy all edge functions
⚠️
Do not expose the service role key under any circumstances. If you find it in a git commit or log file, treat it as compromised and contact Supabase support immediately rather than waiting.

After rotating any secret

  • Update the team password manager with the new value
  • Remove all local copies of the old key (search .env files, terminal history if applicable)
  • Check edge function logs for the first 30 minutes after rotation to confirm no errors
  • Verify that affected functionality still works end-to-end
  • Note the rotation date in the password manager for the next scheduled rotation
💡
When creating new API keys, give them descriptive names that include the environment and year (e.g. communitycarpool-prod-2026). This makes it easy to identify which key is current and when it was created.
