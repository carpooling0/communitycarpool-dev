Organisations

Partner organisations — companies, residential communities, or other groups — can be onboarded to Community Carpool so their employees or members can carpool together. Each organisation gets a unique short code, optional custom branding, named building locations, and configurable matching rules.

What organisations are and why they exist

An organisation record represents a company or community that has partnered with Community Carpool to encourage carpooling among its members. Organisations exist to:

  • Attribute submissions and matches to a specific partner, enabling per-org analytics
  • Show a custom welcome message or logo on the submission form when a user arrives via an org link
  • Offer named building locations (e.g. "Tower A", "HQ Lobby") as quick-select options on the form, making it faster for employees to pick their exact office location
  • Control whether org users can only match within their org or also match with the general public (allow_cross_matching)
  • Generate trackable referral links for distribution to the partner's team

How users join an organisation

Users do not register or "join" an org in a traditional sense. Instead, they arrive at the submission form via a URL that includes an org query parameter:

https://communitycarpool.org/?org=ACME

When the form loads:

  1. The form reads the org query parameter from the URL.
  2. It looks up the org in the organisations table to fetch the display name, logo, and custom welcome message.
  3. If the org has named building locations in org_locations, those appear as quick-select chips in the "From" and "To" location fields.
  4. When the user submits the form, the orgCode value is sent to the submit-journey edge function.
  5. The edge function resolves the org_code to an org_id and stores it on the submissions row.
No opt-in required. There is no separate sign-up flow for an organisation. Any user who arrives via an org link and completes a submission is automatically associated with that org. If a user submits without an org link, their submission has no org affiliation.

The allow_cross_matching flag

Each organisation has an allow_cross_matching boolean (default false) that controls who org users can be matched with:

ValueBehaviourWhen to use
false (default) Org users only match with other users from the same org. They will not appear as matches for public (non-org) users, and public users will not appear as matches for them. Corporate clients who want a private, employees-only pool. Useful when the org has enough volume to sustain intra-org matching.
true Org users are treated like public users for matching purposes — they can match with anyone, including non-org users. Smaller orgs or residential communities where intra-org volume is low and mixing with the public pool increases match quality.
Changing allow_cross_matching only affects future match-finding runs. It does not retroactively create or remove existing matches.

Creating a new organisation

Run the following SQL in Supabase SQL Editor (production project):

-- Step 1: Create the organisation record
INSERT INTO organisations (org_code, org_name, logo_url, custom_message, allow_cross_matching, is_active)
VALUES (
  'ACME',                                        -- short uppercase code, used in URLs
  'Acme Corporation',                            -- full display name shown on form
  'https://acmecorp.com/logo.png',               -- optional: URL to org logo (or NULL)
  'Welcome, Acme team! Find a carpool partner.', -- optional: shown on form (or NULL)
  false,                                         -- false = intra-org matches only
  true                                           -- is_active = true
);

Field reference:

ColumnTypeRequiredNotes
org_codetextYesShort uppercase identifier used in URLs and query params (e.g. EMAAR, ACME). Must be unique. No spaces.
org_nametextYesFull display name shown to users on the submission form and in admin analytics.
logo_urltextNoPublicly accessible URL to the org's logo image. Displayed on the form header when an org is active. Leave NULL if not provided.
custom_messagetextNoShort welcome message shown on the submission form (e.g. "Welcome, Emaar employees!"). Leave NULL to show the default message.
allow_cross_matchingbooleanNoDefaults to false. Set true to allow org users to match with non-org users.
is_activebooleanNoDefaults to true. Set false to deactivate without deleting.
After inserting the org, confirm it was created and note the auto-generated org_id:
SELECT org_id, org_code, org_name, is_active FROM organisations WHERE org_code = 'ACME';

Managing org locations (buildings)

Each org can have any number of named building locations stored in the org_locations table. These appear as quick-select chips in the form's "From" and "To" location fields when a user arrives via that org's link. Clicking a chip pre-fills the location field with that building's coordinates, saving the user from typing an address.

Viewing locations for an org

SELECT l.location_id, l.location_name, l.location_latlng, l.is_active
FROM org_locations l
JOIN organisations o ON o.org_id = l.org_id
WHERE o.org_code = 'ACME'
ORDER BY l.location_name;

Adding a building location

Coordinates must be in "lat,lng" format (decimal degrees). You can get them from Google Maps by right-clicking a location and copying the coordinates.

INSERT INTO org_locations (org_id, location_name, location_latlng, is_active)
VALUES (
  (SELECT org_id FROM organisations WHERE org_code = 'ACME'),
  'HQ Lobby',       -- display name shown on the form chip
  '25.1972,55.2744', -- lat,lng as a text string
  true
);

-- Add more buildings as needed
INSERT INTO org_locations (org_id, location_name, location_latlng, is_active)
VALUES (
  (SELECT org_id FROM organisations WHERE org_code = 'ACME'),
  'Tower A Entrance',
  '25.1985,55.2761',
  true
);

Deactivating a building location

Set is_active = false to hide a location from the form without deleting it:

UPDATE org_locations
SET is_active = false
WHERE location_id = 42; -- use the location_id from the SELECT query above

Reactivating a building location

UPDATE org_locations
SET is_active = true
WHERE location_id = 42;
Coordinate accuracy matters. Building locations are used as the pre-filled "From" or "To" point in the submission. Inaccurate coordinates will cause poor match distances. Always verify coordinates in Google Maps before inserting.

Updating org details

Updating name, logo, or custom message

UPDATE organisations
SET
  org_name       = 'Acme Corporation Ltd',
  logo_url       = 'https://cdn.acmecorp.com/logo-v2.png',
  custom_message = 'Welcome, Acme team! Your carpool partner awaits.'
WHERE org_code = 'ACME';

Removing logo or custom message (set to NULL)

UPDATE organisations
SET logo_url = NULL, custom_message = NULL
WHERE org_code = 'ACME';

Toggling allow_cross_matching

Enable cross-matching for an org that is too small to sustain an intra-org pool:

-- Enable cross-matching (org users match with public pool too)
UPDATE organisations SET allow_cross_matching = true WHERE org_code = 'ACME';

-- Disable cross-matching (org users only match within their org)
UPDATE organisations SET allow_cross_matching = false WHERE org_code = 'ACME';

When to enable it: If an org has fewer than ~30–40 active submissions, intra-org matching alone is unlikely to produce good results. Enabling cross-matching lets those users benefit from the wider public pool while still being tracked as org users in analytics.

Deactivating and reactivating an org

Deactivation prevents the org's landing page from loading (the form will ignore an inactive org code) and excludes the org from new matching logic. It does not delete any data.

Deactivate

UPDATE organisations SET is_active = false WHERE org_code = 'ACME';

Reactivate

UPDATE organisations SET is_active = true WHERE org_code = 'ACME';

What happens to existing data when an org is deactivated

  • Existing submissions: Unaffected. All active submissions from org users remain active and continue to be eligible for matching (subject to allow_cross_matching logic).
  • Existing matches: Unaffected. Confirmed matches between org users are not removed.
  • New submissions via the org link: If a user arrives via ?org=ACME and the org is inactive, the form treats them as a public user — no org branding, no org locations, and no org_id stored on their submission.
  • Building locations: Org locations remain in the database but are not shown on the form while the org is inactive.
Deactivation is reversible. Reactivating an org immediately restores org branding, building locations, and org-scoped matching for any new submissions.

Analytics queries

Submissions per org

SELECT
  o.org_code,
  o.org_name,
  COUNT(s.submission_id) AS submission_count
FROM organisations o
LEFT JOIN submissions s ON s.org_id = o.org_id AND s.journey_status = 'active'
GROUP BY o.org_id, o.org_code, o.org_name
ORDER BY submission_count DESC;

Matches per org (at least one side is from the org)

SELECT
  o.org_code,
  o.org_name,
  COUNT(DISTINCT m.match_id) AS match_count
FROM organisations o
JOIN submissions s ON s.org_id = o.org_id
JOIN matches m ON m.sub_a_id = s.submission_id OR m.sub_b_id = s.submission_id
GROUP BY o.org_id, o.org_code, o.org_name
ORDER BY match_count DESC;

Most active users per org (by submission count)

SELECT
  o.org_code,
  u.name,
  u.email,
  COUNT(s.submission_id) AS submissions
FROM organisations o
JOIN submissions s ON s.org_id = o.org_id
JOIN users u ON u.user_id = s.user_id
WHERE s.journey_status = 'active'
GROUP BY o.org_code, u.user_id, u.name, u.email
ORDER BY o.org_code, submissions DESC;

Org growth over time (submissions per org per month)

SELECT
  o.org_code,
  DATE_TRUNC('month', s.created_at) AS month,
  COUNT(s.submission_id) AS new_submissions
FROM submissions s
JOIN organisations o ON o.org_id = s.org_id
GROUP BY o.org_code, month
ORDER BY o.org_code, month;

Referral links

Generate org-specific links to send to the partner. The org parameter pre-fills the org on the form:

PurposeURL formatEffect
Standard org link https://communitycarpool.org/?org=ORGCODE Loads the form with org branding, building locations, and custom message. Stores org_id on submission.
With UTM tracking https://communitycarpool.org/?org=ORGCODE&utm_source=email&utm_medium=internal&utm_campaign=launch Same as above, plus UTM parameters for campaign analytics in any analytics platform.
QR code for posters Generate a QR code pointing to the standard org link Print on office posters, elevator wraps, or canteen screens.

QR codes can be generated from the Campaign Links panel in the admin panel, which also handles UTM parameters. See Campaign management for details.

Onboarding checklist for a new partner organisation

Use this checklist when onboarding a new partner:

  1. Agree the org_code with the partner (short, uppercase, no spaces — e.g. EMAAR).
  2. Collect the org's full display name, logo URL (publicly hosted), and any custom welcome message.
  3. Decide on allow_cross_matching: false for large corps with their own pool, true for smaller orgs.
  4. Collect named building locations with lat/lng (right-click in Google Maps to get coordinates).
  5. Run the INSERT into organisations (see "Creating a new organisation" above).
  6. Run INSERTs into org_locations for each building (see "Managing org locations" above).
  7. Verify: load https://communitycarpool.org/?org=ORGCODE in a browser and confirm the org name, logo, custom message, and building chips all appear correctly.
  8. Generate the referral link (and QR code if needed) from the admin panel's Campaign Links section.
  9. Send the link/QR code to the partner contact for distribution to their team.
  10. Monitor submissions via the analytics queries above over the first 2–4 weeks to confirm uptake.
