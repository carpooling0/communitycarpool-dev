Security hardening

A complete picture of how Community Carpool is secured at every layer — database, API, admin access, email, and credentials — plus known gaps and the steps to address them.

Security model overview

The platform has no traditional server or VPC to protect. Instead, security is enforced at three distinct boundaries:

BoundaryMechanismWhat it protects
Database (Supabase)Row Level Security (RLS) with no policiesDirect table access — anon and authenticated roles are completely blocked. Only the service role key bypasses RLS.
API (Edge Functions)Session tokens, JWT verification, input validationAll write and read operations on user data. Each function validates the caller before touching the DB.
Admin panelPBKDF2 password hashing, session tokens, role gates, optional IP allowlist, deletion PINAll admin operations. No direct DB access for Admin or Support roles.

Database security — Row Level Security (RLS)

How RLS works here

Supabase exposes three database roles: anon (unauthenticated browser requests), authenticated (signed-in Supabase Auth users — not used here), and service_role (edge functions, Supabase dashboard).

When RLS is enabled with zero policies, the default PostgreSQL behaviour is deny all for anon and authenticated. The service_role bypasses RLS entirely. This means:

  • Any browser hitting the Supabase REST API with the anon key gets an empty result or permission denied — even if they know the table name
  • Edge functions (service role) continue to work normally
  • No explicit deny policies are needed — absence of allow policies achieves the same result
⚠️
The anon key is public. It is shipped in config.js and visible in any browser's DevTools. It is designed to be public — security relies entirely on RLS blocking what the anon role can do, not on keeping the key secret.

Verifying RLS in Supabase

To confirm RLS status at any time, run this query in the Supabase SQL editor:

SELECT tablename, rowsecurity AS rls_enabled,
  (SELECT COUNT(*) FROM pg_policies p
   WHERE p.tablename = t.tablename AND p.schemaname = 'public') AS policy_count
FROM pg_tables t
WHERE t.schemaname = 'public'
ORDER BY tablename;

Every table should show rls_enabled = true. If a new table is created via migration and RLS is not explicitly enabled, it will default to false — always include ALTER TABLE <name> ENABLE ROW LEVEL SECURITY; in every migration that creates a table.

💡
Supabase Security Advisor (Dashboard → Advisors → Security) flags tables with RLS disabled. Check it after every migration. The only expected flag is spatial_ref_sys which cannot be fixed — suppress it if possible.

Edge function security

JWT verification

Each edge function has a verify_jwt setting in its config. Functions that are callable from the browser (user-facing) use verify_jwt: true, which means Supabase validates that the Authorization header contains a valid, unexpired anon JWT before the function code even runs. Functions that should only be called by cron jobs use verify_jwt: false and implement their own caller validation internally.

CORS

All edge functions return CORS headers that restrict which origins can call them. The allowed origin is set to https://communitycarpool.org (prod) or https://dev.communitycarpool.org (dev). A random origin cannot call the functions from a different website.

Check CORS config in the function source: look for Access-Control-Allow-Origin in the response headers.

Input validation

Edge functions validate all required fields before touching the database. Key checks in place:

  • submit-journey — validates email format, required fields, checks max_active_submissions config, validates org_code if provided
  • submit-support-ticket — rejects type: "deletion" (must use the manage-deletion flow), validates email, subject, message
  • manage-deletion — validates token exists and is unexpired; unregistered emails silently return 200 to avoid email enumeration
  • update-match-status — validates that the requesting user actually owns the submission involved in the match before allowing interest/decline
  • admin-api — validates session token on every request; enforces role-based access per action before executing

Token-based user authentication

CCP does not use Supabase Auth or passwords for end users. Authentication is token-based:

  • When a match is found, the user receives a unique match_page_token in an email link
  • The token is stored (hashed in future; currently plain) in the users table as match_page_token
  • Tokens expire after match_token_expiry_days (currently 120 days) — configurable in the config table
  • get-matches-page validates the token on every call; expired tokens return 401
  • If a user's account is deleted, the token is also deleted — the link becomes permanently dead
⚠️
Future improvement: The match_page_token is currently stored as plain text in the DB. It should be hashed (e.g. SHA-256) so that a DB read does not reveal usable tokens. Low urgency given RLS protects the table, but worth tracking.

Admin panel security

Authentication

Admin users authenticate via the admin-auth edge function. Login checks run in this order:

  1. Email exists in admin_users table and is_active = true → else 401
  2. PBKDF2 password verify (16-byte salt, 32-byte key, 100,000 iterations, SHA-256) → else 401
  3. role_expires_at is set and now() > role_expires_at → 403 "Access expired"
  4. allowed_ips is set and request IP not in list → 403 "IP not allowed"
  5. Session token created (32-byte random hex), stored in admin_sessions with 8-hour expiry

Role-based access

Three roles control what each admin can see and do:

RolePanels accessibleDestructive actions
super_adminAll (Dashboard, Tickets, Deletions, Users, Campaigns, Audit Log, Team)User deletion (PIN required), team management, all admin operations
adminDashboard, Tickets, Deletions, Users, CampaignsUser deletion (PIN required), blacklist, cancel deletions
supportDashboard, Tickets onlyNone — read + close tickets only

Role checks happen server-side in admin-api on every request — the frontend role-gating is UI convenience only and cannot be bypassed to gain elevated access.

Deletion PIN

Permanent user deletion requires a 6-digit PIN separate from the login password. The PIN hash is stored in admin_users.deletion_pin_hash using the same PBKDF2 algorithm. This means:

  • Knowing an admin's password does not grant deletion capability
  • A stolen browser session cannot delete users without the PIN
  • PINs are set at account creation and can be reset by a Super-Admin

IP allowlisting

The allowed_ips column on admin_users accepts an array of permitted IP addresses. When set, login is rejected from any other IP with a 403. This is optional per admin — leave NULL to allow any IP. Recommended for Super-Admin accounts where the admin has a fixed IP.

Role expiry

The role_expires_at column automatically locks out an admin after a date without needing manual deactivation. Useful for temporary access grants or contractors. After the date, login returns 403 — existing sessions already in use will fail on the next API call.

Session management

Sessions are stored in the admin_sessions table with a 8-hour expiry. The session token is a 32-byte random hex string stored in localStorage in the browser. On logout, the session row is deleted. On expiry, the next API call returns 401 and the browser redirects to the login screen.

Email security — SPF, DKIM, DMARC

Email authentication prevents spoofing of the communitycarpool.org domain. See DNS records for exact record values.

RecordStatusWhat it does
SPF on send.communitycarpool.org Active Authorises AWS SES to send email from the send. subdomain. Without SPF, receiving servers may mark SES mail as spam.
DKIM (3 CNAMEs) for SES Active Cryptographically signs every outbound SES email. Allows receiving servers to verify the message was not tampered with in transit.
DKIM for Resend Active Resend DKIM configured on the root domain for Resend-sent emails.
MX on send.communitycarpool.org Active Routes SES bounce/complaint feedback to Amazon's feedback processing endpoint.
DMARC on communitycarpool.org Monitoring only Current policy: p=none — reports are generated but no enforcement action is taken. Spoofed emails will still reach inboxes. See upgrade path below.

DMARC upgrade path

The current p=none policy is a monitoring stance. To enforce it:

  1. Add a rua= address — point DMARC aggregate reports to a mailbox you check (e.g. rua=mailto:carpooling0@gmail.com). This lets you see whether any legitimate mail is failing DMARC before you tighten the policy.
  2. Upgrade to p=quarantine — failing emails go to the spam folder instead of the inbox. Monitor for 2–4 weeks.
  3. Upgrade to p=reject — failing emails are outright rejected. This is the gold standard and prevents all domain spoofing.
⚠️
Do not jump straight to p=reject without monitoring first. If any legitimate sending path (Resend, SES, or a third-party tool like Google Workspace) is not correctly DKIM-signed and SPF-aligned, those emails will be rejected by the receiving server.

⚠️
Never commit secrets to git. The repositories are public. Run git log --all --full-diff -p -- config.js periodically to verify no secrets have been accidentally committed. If a secret is ever found in git history, treat it as compromised and rotate immediately — deleting the commit does not remove it from forks or caches.

GitHub & deployment security

  • Public repositories — both carpooling0/communitycarpool and carpooling0/communitycarpool-dev are public. Never push anything that contains a secret, personal data, or internal credentials.
  • Two-remote discipline — always push to dev-repo (dev) unless explicitly deploying to production. Pushing to origin (prod) immediately goes live.
  • No branch protection on main — force push is possible. Be careful. Consider enabling branch protection on the prod repo.
  • GitHub Actions not used — deployment is fully manual (push to repo → GitHub Pages rebuilds). There are no CI secrets or workflow files to audit.
  • super-admin docs — currently excluded from the production GitHub repo. They are served only from the dev environment. Do not push them to the prod repo until this policy is reviewed.

Security checklist — new team member onboarding

  • ☐Receive credentials via team password manager only — never via email or chat
  • ☐Confirm which Supabase project is dev vs prod before running any SQL or deploying functions
  • ☐Never push to origin (prod GitHub repo) unless deploying an approved change
  • ☐Never copy secrets into code, config files, or commit messages
  • ☐Do not share admin panel credentials — each team member has their own admin_users row
  • ☐Review the Supabase Security Advisor after any migration to check for new RLS gaps
  • ☐When adding a new table: always include ALTER TABLE <name> ENABLE ROW LEVEL SECURITY; in the migration

Security checklist — after any team member offboarding

  • ☐Set is_active = false on their admin_users row immediately
  • ☐Delete any active rows in admin_sessions for that admin_id
  • ☐Rotate any secrets they had access to (see Rotating secrets)
  • ☐Remove them from the team password manager
  • ☐Revoke their GitHub access if they were a repo collaborator
  • ☐Rotate Porkbun account password if they had DNS access

Security checklist — after adding a new table

  • ☐Migration includes ALTER TABLE <name> ENABLE ROW LEVEL SECURITY;
  • ☐Run the RLS status query above to confirm rls_enabled = true
  • ☐Check Supabase Security Advisor — no new unresolved warnings
  • ☐Document the table in Tables reference
